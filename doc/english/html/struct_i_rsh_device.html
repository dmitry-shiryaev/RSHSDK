Detailed Description

Interface is used to control almost all RSH ADC/DAC devices.

Main interface to handle ADC and DAC devices produced by CJSC "Rudnev-Shilyaev" and some third-party devices by other companies.
Typical scenario is when user gets object with IRshDevice interface from device abstraction library, which can be loaded using RshDllClient class. Then user can get access to all features of the device without knowing low level architecture of device and device drivers.

Remarks
Some RSh devices do not use this interface, for example, charge amplifiers that are connected to PC using seral port and accessed via RS-232 interface.

virtual U32 __stdcall IRshDevice::Connect ( RshBaseType *  key,
U32  mode = RSH_CONNECT_MODE_BASE 
)
pure virtual

Connect to certain device.

Parameters
[in]keyIdentification key for a device. It might be an index of the device, a serial number of the device or a connection string to the device.
[in]modeConnection mode using a device index, a serial number or a connection string.
Returns
RSH_API_SUCCESS or error code.

This method must be executed first before any other device access methods.
After successfull call of this method device is ready to use and can be configured for data acquisition.
Most other methods of interface (such as Init(), Start() etc.) will return error if device was not connected using Connect().
One can, however, get some static library and device information using Get() method.

Remarks
If you use connection mode RSH_CONNECT_MODE_BASE devices are enumerated from 1. Order of devices is determined by operating system and depends on used slot and plug order.

virtual U32 __stdcall IRshDevice::GetData ( RshBaseType *  buffer,
U32  flags = RSH_DATA_MODE_NO_FLAGS 
)
pure virtual

Get (or send) data buffer to device.

Parameters
[in,out]bufferBuffer for (with) data. One of the RshBufferType simple type classes.
RSH_BUFFER_S8, RSH_BUFFER_S16, RSH_BUFFER_S32 are used in most cases for transfer converted by ADC data. Data is always shifted so MSB will be first (left) bit.
RSH_BUFFER_DOUBLE can be used too, for transfer data converted to volts.
User must allocate memory for data he wants to transfer before GetData() Call! One can do so with RshBufferType::Allocate method.
[in]flagsAdditional flags for data conversation. Can be used, for example, to get acces to digital lines at the same time with data acquisition and write this information with data. Flag list: RSH_DATA_MODES.
Returns
RSH_API_SUCCESS or error code.

This method is used to transfer data from (or to) device abstraction library. After receiving RSH_GET_WAIT_BUFFER_READY_EVENT data is ready for transfer. One must allocate buffer. If allocated buffer size is less than total data available, buffer will be filled with data that fit into it, no error will be thrown. If the opposite true, its no problem also. One can check actual data transfered to buffer looking to RshBufferType::size value.

Remarks
When selecting buffer size in initialization structure its always per channel value. So, for example, one set buffer size in initialization structure to 2048 and number of channels to 2, one must expect total data available in GetData() method to be equal to 2*2048 = 4096 samples.

virtual U32 __stdcall IRshDevice::Init ( RshBaseType *  structure,
U32  mode = RSH_INIT_MODE_INIT 
)
pure virtual

Setup data acquisition parameters.

Parameters
[in,out]structureInitialization structure derived from RshBaseType.
Each device can support one or more of initialization structures, such as RshInitDMA or RshInitMemory.
[in]modeInitialization mode:
RSH_INIT_MODE_CHECK - check and change wrong parameters in structure without any hardware access.
RSH_INIT_MODE_INIT - real initialization, check and change wrong parameters and send them to hardware.
Returns
RSH_API_SUCCESS or error code.

This method is used to setup device and set data acquisition parameters. Special parameter structures are used:
RshInitDMA: for low frequency (less than 10MHz) devices with no integrated flash memory.
RshInitMemory: for high frequency devices with on board memory.
RshInitGSPF: for digital generator devices.
RshInitVoltmeter: for B7/78 voltmeter.
RshInitPort: access to digital port.
RshInitDAC: access to DACs.
Using these structures one can set various settings such as buffer size, frequency, number of channels and so on. See structure description for more details. Example:

IRshDevice* device;
//initialization structure
RshInitDMA params;
params.startType = RshInitDMA::Program; //start from internal timer
params.frequency = 10000.0; //sampling rate (per channel)
params.dmaMode = RshInitDMA::Single; //single start
params.bufferSize = 2048; //buffer size (per channel)
//channels setup
//set channel 0 as active
params.channels[0]SetUsed();
//Alternativly one can manipulate bits in channel's control word directly
//params.channels[0].control |= RshChannel::Used;
U32 res = device->Init(&params);
if(res != RSH_API_SUCESS)
{
//process error
}
Remarks
You can find what init structure is supported by your device using Get() method with parameter RSH_GET_DEVICE_IS_CAPABLE and check capabilities RSH_CAPS_SOFT_INIT_MEMORY, RSH_CAPS_SOFT_INIT_DMA etc.

virtual U32 __stdcall IRshDevice::Start ( )
pure virtual

Start data acquisition or signal generation process.

Returns
RSH_API_SUCCESS or error code.

ADC devices start data acquisition and transfer when this method is called. After data transfer complete, data ready event will be signalled and data can be collected using GetData() method.
Signal generators (GSPF family for instance) start buffer translation on this command. So, for them command order is reverced: one need first to transfer data buffer using GetData() method and only after that start generation with Start().

Remarks
Process of starting device is not immidiate, some time required for transfer command and process it.
Moreover, if synchronization of some kind used, device only start actual data acquisition process when synchronization event will be triggered.

virtual U32 __stdcall IRshDevice::Stop ( )
pure virtual

Stop data acquisition or signal generation process.

Returns
RSH_API_SUCCESS or error code.

When this method is called, all data connected activity is terminated: continious data transfer is cancelled, ADC (or DAC) is stopped, USB pipes and DMA channels are reseted to default state and ready to next Start() call.

Remarks
If this method is called before any actual data transfer commited, Get() method with RSH_GET_WAIT_BUFFER_READY_EVENT parameter will return error code as result.
